Buttons

Buttons allow users to trigger actions or navigate to another destination. The mngha button component supports three sizes, eight visual variants, disabled states, links styled as buttons, and optional leading or trailing icons.

View Components

Overview

Composable classes

Combine the base mngha-button class with one size modifier and one visual variant.

Theme-aware actions

Primary, outline, secondary, and ghost buttons use semantic theme tokens and adapt to the active external or internal theme.

Native semantics

Use a real button for actions and an anchor styled as a button only when the control navigates to another page or resource.

Basic Button

Every button requires the base class, one visual variant, and one size modifier.

Example

HTML

Basic button

<button
  class="mngha-button mngha-button--primary mngha-button--md"
  type="button"
>
  Continue
</button>

Button Variants

Select a variant according to the action's importance and meaning. Avoid using several visually dominant actions in the same group.

Primary

Use for the main action on a page or inside a focused task.

Secondary

Use for a supporting action that should remain visually prominent.

Outline

Use for an alternative action with a lighter visual weight.

Ghost

Use for low-emphasis actions placed near related content.

Danger

Use for destructive or irreversible actions.

Warning

Use for actions requiring caution or confirmation.

Success

Use for positive completion or confirmation actions.

Info

Use for informational actions or supporting details.

Button and Link Elements

Match the HTML element to the behavior. Buttons perform actions; anchors navigate to another destination.

Action button

Use a native button for submitting, saving, opening a dialog, toggling content, or changing application state.

Navigation link

Use an anchor with button classes when the destination has a URL and should support normal link behavior.

Disabled States

Use the native disabled attribute on button elements. For links that must appear unavailable, add aria-disabled="true", remove the destination, and prevent activation in application logic.

Accessibility

Use the correct element

Use button elements for actions and anchor elements for navigation. Do not use generic div or span elements as buttons.

Write clear labels

Describe the action directly with labels such as Save changes, Download report, or Delete account.

Preserve focus

Do not remove the component's focus-visible outline. Keyboard users need a clear indication of the active control.

Label icon-only controls

When a button contains no visible text, provide a descriptive aria-label and hide decorative icons from assistive technology.

Do not rely on color

The visible label must explain the action. Color alone should not be the only indication of danger, success, or warning.

Use disabled states carefully

When possible, explain why an action is unavailable instead of leaving users with an unexplained disabled control.

Usage Guidance

Do

Use one primary action in a focused area, keep labels concise, and use consistent sizes inside the same action group.

Avoid

Do not place several primary, danger, or warning buttons together unless every action genuinely has equal importance.

Match meaning to variant

Reserve semantic variants for actions with that meaning. Do not use danger or success colors only for decoration.

Keep text readable

Prefer short verb-led labels and avoid wrapping long sentences inside buttons.
